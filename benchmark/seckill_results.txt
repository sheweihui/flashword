========================================
  四、数据分析
========================================

1. 响应时间分析
   - P50 仅 53ms：大部分请求被 Redis 快速拦截（限流/重复）
   - P95 平均 201ms：成功抢购涉及 MySQL + MQ，耗时较长
   - 三轮差异说明：随轮次增加，MQ 队列积压、连接池竞争导致
     响应时间上升

2. 拦截效果
   - 限流拦截（用户级 3 req/s + 全局令牌桶 200/s）优先生效
   - 一人一单（SETNX）拦截剩余重复请求
   - 86.4% 的请求被提前拦截，避免无效写入数据库
   - 真正走到下单的仅 200 个，全部成功

3. 系统瓶颈
   - 全局限流是最直接的限制因素，QPS 被控制在 200/s 以内
   - MySQL 连接池 50、Redis 连接池 50 在当前规模下够用
   - MQ 消费者处理能力（5-10 并发）在持续压力下出现积压

4. 正确性验证
   - 三轮均精确售出 200 单，无超卖
   - Redis（4800）与 MySQL（4800）库存严格一致
   - HTTP 失败率 0%，无数据丢失

========================================
  五、结论
========================================

秒杀接口在高并发场景下表现稳定:
  ✅ 响应快: P95 平均 201ms，过半数请求 < 53ms
  ✅ 不超卖: Redis DECR + MySQL 乐观锁双重保障
  ✅ 防刷单: 限流 + 一人一单拦截 86% 无效请求
  ✅ 高可靠: 0% HTTP 错误率，200 单全部正确完成

优化建议:
  - 如需更高 QPS，调整令牌桶容量（当前 200/s）
  - 考虑秒杀商品使用独立 Redis 实例避免资源争抢
  - 增加 MQ 消费者并发数以消除积压
